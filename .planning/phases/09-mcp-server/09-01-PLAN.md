<objective>
Make Relay's dispatch model-agnostic: any OpenAI-compatible (or Anthropic-messages) endpoint becomes a usable provider through env config alone, with the proven registry pattern ported from the sunsetted relay-mcp predecessor. Kill the closed provider unions in the run path. Builtin providers keep working unchanged; dynamic providers ride the existing GenericHttpRunner. Run records keep storing the raw provider usage as a receipt — no price math.

Purpose: DISPATCH-01 is the core of the v0.4 scope ("talk to anything that has an LLM model"). Today `src/cli/cmd-run.ts` hardwires five providers behind a closed union; adding a sixth means a code change. relay-mcp already solved this with `RELAY_PROVIDER_<NAME>_*` env discovery (`config/providers.ts` + `registry.ts` in /Users/ghanavati/ai-stack/Projects/relay-mcp — read-only reference, port the pattern, not the code). v0.4 ports the lean subset: URL, KEY, TYPE (protocol adapter), HEADER_*. The adapter zoo (ADAPTER_TYPE, OPENCLAW, EXECUTABLE, INTEGRATION_LEVEL) stays dead — it served relay-mcp's scope sprawl.

Output: `src/workers/provider-registry.ts`, a parameterized GenericHttpRunner, a registry-resolving cmd-run, and `relay providers`.

Scope decisions baked in (see 09-CONTEXT.md):
- TYPE enum v1 = `openai` (chat-completions wire, the default) | `anthropic` (messages wire). `openai-responses` is deferred — OpenAI still serves chat/completions and nearly every third party is chat-completions-compatible.
- Dynamic providers are single-shot (non-agentic) in v1. The agentic tool_loop stays bound to lmstudio-agentic; generalizing it is a later, measured step.
- Builtin names win: an env definition colliding with a builtin name is a RelayError at resolve time, not a silent override.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User env → outbound HTTP | Env-declared URLs receive the task content and an API key; a typo'd URL sends prompt + key to the wrong host. |
| Provider config → terminal/logs | Config listing and error paths must never print key values. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Information Disclosure | API key in `relay providers` / errors / run records | mitigate | Configs carry keyEnvVar names only (Task 1); value resolved at request time inside the runner; existing redaction covers error/spawn paths; Task 3 Test 4 asserts masked output. |
| T-09-02 | Spoofing | builtin provider silently overridden by env | mitigate | Builtin-wins + RelayError on name collision (Task 1, Test 6). |
| T-09-03 | Information Disclosure | task content + key sent to user-mistyped URL | accept | The env var is user-controlled local config — same trust as editing a config file. The providers listing makes the resolved URL visible for self-audit. |
</threat_model>
