<objective>
Expose Relay's one genuinely unique capability — persistent cross-session, cross-tool memory — over MCP as two tools: `relay_memory_recall` (token-budgeted recall for the current workdir) and `relay_memory_save`. Both are thin wrappers over the existing memory handlers, reusing the existing Zod schemas and the same SQLite store + workdir scoping the CLI uses.

Purpose: these two tools ARE the v0.4 MCP surface (D-02 in 09-CONTEXT.md — the killed session-control tools do not ship). MCP-02 requires memory tools over MCP to operate on the same memory store + workdir scoping (`RELAY_MEMORY_ALLOWED_WORKDIRS`) as the CLI. MCP-03 requires the inputSchema to BE the contracts schemas. The memory handlers `handleRecall`/`handleRemember` already exist and already emit the `{ content }` envelope, but they do NOT redact — MCP-04 requires redaction at the boundary, so the wrappers route through the result.ts helpers (Plan 02). Workdir scoping is enforced inside `MemoryStore.assertWorkdirAllowed`, so wrapping the handlers (not bypassing them) gives scoping for free; the test must prove a forbidden workdir surfaces as an MCP error, not a silent cross-project read.

Output: `src/mcp/tools-memory.ts` exporting `buildMemoryMcpTools()` — the registrations the server (Plan 04) hands to `registerTool`.

Consent note (formerly O-03, now decided): `relay_memory_save` ships under the SAME workdir-scoping gate the CLI uses — no looser path, no separate consent ceremony. The memory layer's existing write protections (rate limit per source run, 60s dedup, redaction-on-save) apply unchanged because the wrapper calls the same store path.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → Relay memory store | An external client reads and writes Relay's persistent memory; cross-project reads or unscoped writes would leak or pollute other workdirs' memory. |
| Relay memory store → MCP client | Recalled memory text may contain secrets a prior session stored. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-08 | Information Disclosure | relay_memory_recall cross-workdir read | mitigate | Tool calls through MemoryStore whose assertWorkdirAllowed enforces RELAY_MEMORY_ALLOWED_WORKDIRS (Task 1, Test 3); forbidden workdir → isError, never another project's memory. |
| T-09-09 | Information Disclosure | recalled memory text leaking secrets | mitigate | Boundary redaction on the returned text (Task 1, Test 4) — handleRecall does not redact, the MCP wrapper does. Store-side redaction-on-save is a second layer, not a substitute. |
| T-09-10 | Tampering | unscoped or runaway memory writes from an MCP client | mitigate | relay_memory_save inherits the same MemoryStore gates: workdir scoping (Task 2, Test 3), per-source write rate limit, 60s content dedup, unverified-by-default trust for non-human sources. |
</threat_model>
