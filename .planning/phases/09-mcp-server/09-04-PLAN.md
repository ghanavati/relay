<objective>
Make an MCP caller a first-class, policy-bound control session and expose the six session-control tools over MCP by wrapping the existing `src/control/tools.ts` handlers. The MCP connection registers an llm-kind control session on connect (caller id bound server-side), the tools run as that caller through the SAME broker the agentic path uses, and the session ends on disconnect.

Purpose: This exposes the six session-control tools named in D-03 by REUSING src/control/tools.ts (D-06 confinement to src/mcp/). MCP-03 requires reusing the existing control handlers (the server registers them, does not reimplement). MCP-04 requires an MCP caller to be an llm-kind control session under the same broker policy as the agentic path: default-deny cross-session sends, grants with TTL + budget, loop detection, SQLite audit events, and no path to self-escalate. The `relay run` agentic path already models this exactly — `createControlSessionForRun` + `registerControlTools(run_id)` + `endControlSessionForRun` (see src/cli/cmd-run.ts). This plan mirrors that pattern for an MCP connection, which is the structural guarantee against the headline threat: an MCP client sending cross-session messages as if it were human.

Output: `src/mcp/session.ts` (`createMcpControlSession`) and `src/mcp/tools-control.ts` (`buildControlMcpTools(callerSessionId)`), wired in Plan 05's server.

This plan contains a maintainer decision (O-02) — see Task 1.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → control broker | An external MCP client drives session-control tools. If it could send as a different source or without a grant, it would bypass Phase 8 policy and message arbitrary sessions as if human-authorized. |
| MCP connection lifecycle → session store | A connection registers/ends a control session; a leaked or reused id could let one client act as another. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-09 | Spoofing | MCP caller sending as another session / as human | mitigate | source_session_id bound server-side at registration (registerControlTools(callerId)); strict schema rejects a model-supplied source key (Task 3, Test 2). Caller id derived in createMcpControlSession, never from args. |
| T-09-10 | Elevation of Privilege | MCP caller bypassing default-deny / self-escalating | mitigate | All sends route through the existing broker (sender_kind 'llm' → default-deny); no grant/approve tool is exposed (Task 3, Tests 3/6). Loop + self-send + budget enforced by the reused broker (Test 5). |
| T-09-11 | Repudiation | untraceable MCP-driven actions | mitigate | Session registered/ended with audit events (Task 2); every send/block emits SQLite audit events anchored per Phase 8 D-05 (Task 3, Tests 3/4). |
| T-09-12 | Information Disclosure | session content/ids leaking secrets | mitigate | Results redacted at the MCP boundary (Task 3, Test 7). |
</threat_model>
