<objective>
Assemble the stdio MCP server and wire it to a new `relay mcp` command. On start the server resolves the verified SDK surface, registers the two memory tools, and connects over stdio. The mcp path is stdout-disciplined: stdout carries only the MCP protocol; everything else goes to stderr.

Purpose: MCP-01 requires `relay mcp` to start a stdio server an MCP client registers via `.mcp.json` and whose tools it can enumerate. MCP-05's closing clause requires the SDK imports to be verified against the installed version (via Plan 02's resolveMcpSdk) and the existing CLI surface to stay unchanged (additive only). The stdout-corruption risk is the operational landmine of stdio MCP — a single `console.log` to stdout breaks the framing — so this plan tests that the mcp path emits nothing to stdout outside the protocol.

This server is deliberately thin. The earlier draft bound each MCP connection to an llm-kind control session and registered six session-control tools; that scope is KILLED in v0.4 (see 09-CONTEXT.md). No control session is created, no broker is touched, no dispatch/shell surface exists over MCP. A memory recall/save server needs none of that machinery — provenance is covered by the memory source tag and the store's own gates (Plan 03).

Output: `src/mcp/server.ts` (`startMcpServer`), `src/cli/cmd-mcp.ts` (`executeMcpCommand`), and an `mcp` branch in `src/cli.ts`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| stdio wire ↔ MCP client | The server speaks MCP framing on stdout/stdin; any non-protocol byte on stdout corrupts the stream and can desync or crash the client. |
| local process → relay mcp stdin | stdio MCP trusts the launching parent: any local process that can spawn `relay mcp` acts with the OS user's authority — the same trust boundary as the CLI itself. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-11 | Tampering | stray stdout write corrupting the MCP wire | mitigate | server.ts + cmd-mcp.ts write nothing to stdout (grep gate + Task 2 Test 1 asserts diagnostics land on stderr, stdout stays protocol-only). |
| T-09-12 | Denial of Service | SDK import surface mismatch at connect | mitigate | startMcpServer builds from resolveMcpSdk (Plan 02), which verified the installed entry points (Task 1, Test 3). |
| T-09-13 | Elevation of Privilege | dispatch/shell/control surface exposed over MCP | mitigate (by absence) | The killed scope is structurally absent: server.ts imports no broker/control/run module (Task 1 grep gate); only the two memory tools register (Test 1). |
| T-09-14 | Spoofing | local-process access to the stdio server | accept | stdio inherits the OS-user trust boundary — identical to running the relay CLI. Remote/multi-client transports (with real auth) are explicitly v2 scope. |
</threat_model>
