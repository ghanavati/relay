<objective>
Assemble the stdio MCP server and wire it to a new `relay mcp` command. On start the server resolves the verified SDK surface, registers the memory tools and the session-control tools (bound to a per-connection llm-kind control session), connects over stdio, and tears the session down on exit. The mcp path is stdout-disciplined: stdout carries only the MCP protocol; everything else goes to stderr.

Purpose: This assembles the stdio-only transport chosen in D-01 inside the src/mcp/ module + a thin src/cli.ts branch per D-06. MCP-01 requires `relay mcp` to start a stdio server an MCP client registers via `.mcp.json` and whose tools it can enumerate. MCP-07 completion requires the SDK imports to be verified against the installed version (via Plan 01's resolveMcpSdk) and the existing CLI surface to stay unchanged (additive only). The stdout-corruption risk is the operational landmine of stdio MCP — a single `console.log` to stdout breaks the framing — so this plan adds a test that asserts the mcp path emits nothing to stdout outside the protocol.

Output: `src/mcp/server.ts` (`startMcpServer`), `src/cli/cmd-mcp.ts` (`executeMcpCommand`), and a `mcp` branch in `src/cli.ts`.

Open-question note for the maintainer (O-01 — `relay run`/dispatch over MCP): this plan registers ONLY the memory + session-control tools. It does NOT expose worker dispatch / `relay run` / `shell_exec` over MCP (the D-04 default: deferred — it is side-effecting and deserves its own consent design). If the maintainer wants dispatch over MCP in v1, that is a separate plan; flagged here, not silently included.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| stdio wire ↔ MCP client | The server speaks MCP framing on stdout/stdin; any non-protocol byte on stdout corrupts the stream and can desync or crash the client. |
| process lifecycle → control session | The connection's control session must end on disconnect/exit or it lingers as an active session with any grants attached. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-13 | Tampering | stray stdout write corrupting the MCP wire | mitigate | server.ts + cmd-mcp.ts write nothing to stdout (grep gate + Task 2 Test 1 asserts diagnostics land on stderr, stdout stays protocol-only). D-11 stdout discipline. |
| T-09-14 | Denial of Service | SDK import surface mismatch at connect | mitigate | startMcpServer builds from resolveMcpSdk (Plan 01), which already verified the installed entry points (Task 1, Test 5). |
| T-09-15 | Repudiation | leaked/lingering control session after disconnect | mitigate | Session end() wired to transport close (Task 1, Test 3); SIGINT/SIGTERM teardown in cmd-mcp (Task 2). |
| T-09-16 | Elevation of Privilege | dispatch/shell_exec exposed over MCP without consent | accept (deferred) | O-01 default: no dispatch tool registered (Task 1 — only memory + control tools). Worker dispatch over MCP is out of scope for v1 pending maintainer decision. |
</threat_model>
