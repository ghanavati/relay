# Relay Memory Fixtures

Binary SQLite fixtures used by the Phase 1 schema-cleanup tests and any
future schema-migration work.

## Files

| File | Purpose | Size cap |
| ---- | ------- | -------- |
| `v0.1.2-baseline.db` | Pre-v0.2 database shape (11 orphan tables populated, 3 memory rows). Used by `src/runtime/store/migrate-v2-drop-orphans.test.ts` to prove the v2 cutover is lossless. | ≤30 KB |
| `_generate-v0.1.2.mjs` | Deterministic regenerator for `v0.1.2-baseline.db`. Inlines DDL from git tag `v0.1.2` (commit `7c7afc2`). | — |

## Provenance

`v0.1.2-baseline.db` was generated by `_generate-v0.1.2.mjs`, which inlines the
DDL of git tag `v0.1.2` (commit `7c7afc2`). The script anchors all timestamps
to `1700000000000` (2023-11-14T22:13:20Z) and uses deterministic primary keys
(`fix-…`) so re-running yields a near-byte-stable file. A `page_size = 512`
PRAGMA keeps the fixture compact.

## Regen procedure

```bash
node src/memory/__fixtures__/_generate-v0.1.2.mjs
git diff --stat src/memory/__fixtures__/v0.1.2-baseline.db
```

If the diff churns more than a handful of pages, investigate before committing
— SQLite's internal layout can shift if the script's row order or pragma set
changes. Stable regen is part of the contract; tests do not check byte
equality but reviewers should be able to diff this binary deterministically.

## Anonymisation contract

Any row added to this fixture MUST be anonymised:

- No real API tokens, no PATs, no `sk-…` literals, no `Bearer …` headers.
- No real user names, real `/Users/...` paths, no real workdirs.
- Use deterministic placeholder strings (`fixture-…`, `[anonymized …]`).
- Paths use `/fixture/workdir` exclusively.

When updating the generator, re-read this checklist and verify each new row
satisfies it before regenerating. Reviewers will reject diffs that introduce
real-looking secrets, even in committed binaries.

## Why a binary fixture?

The CC.1 enforcement gate in `.planning/phases/01-schema-cleanup/PLAN.md`
(PITFALLS.md:233-239) requires us to prove the v2 migration is lossless
against an actual v0.1.2-shaped database. A synthetic in-memory schema would
not catch issues the generator doesn't model (e.g. autoincrement values,
trigger fire counts, FK constraint timing). A committed fixture is the
cheapest way to make every test run hit the real layout.
