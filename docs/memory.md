# Memory

Relay has a persistent memory store — a local SQLite table where you save facts, lessons, decisions, and notes that survive across sessions.

## Why

Claude Code sessions reset. Codex CLI invocations are stateless. Without persistent memory, you re-explain context every session. Relay's memory store + the recalled_lessons context layer = automatic injection of relevant past lessons into new sessions.

## Memory types

The `memory_type` field categorizes entries:

- `fact` (default) — stable, unchanging knowledge: API endpoints, file paths, version constraints
- `decision` — chosen approach with rationale: "we use SQLite over Postgres because solo deployment"
- `lesson` — something learned from a failure or success: "never run rm -rf with absolute paths via the Bash tool"
- `context` — background info about a project, design, or domain
- `state` — a snapshot of progress at a point in time (rare in solo use)
- `handoff` — session-to-session continuity packet (auto-generated by `record-session-thinking` if used)

The show-context CLI defaults to `lesson` + `decision` (the high-signal types for new sessions).

## Lifecycle

Writing:
```
relay memory remember 'Berry verifier needs gpt-4.1-nano direct OpenAI' --type lesson
```

Reading:
```
relay memory recall 'berry'
```

The recall query uses FTS5 + recency fallback. If the query matches no entry by full-text, the most recent N entries are returned (capped by `--token-budget`).

Garbage collection (auto):
- On every `remember()` write, `gcByTokenBudget()` runs. If total tokens > `RELAY_MAX_MEMORY_TOKENS` (default 100000), oldest non-pinned entries are evicted.
- `purgeSuperseded()` runs inside `upsert()` to hard-delete tombstones older than 30 days.

Garbage collection (manual):
```
relay memory gc --max-age-days 30
```
Soft-deletes pinned entries not accessed in N days. Pinned entries with no expiry can persist indefinitely otherwise.

## Trust levels

Relay computes a trust level for each entry on read:

| Level | When |
|---|---|
| `trusted` | `memory_source: 'human'` AND `pinned: true` — manually saved + pinned |
| `trusted` | `success_recall_count >= 3` — recalled successfully 3+ times |
| `provisional` | human source OR 1+ successful recall |
| `unverified` | default for auto-written entries |

## The recalled_lessons context layer

When Relay dispatches a task to a worker (codex, lmstudio, openrouter), the recalled_lessons context layer is injected if `RELAY_RECALLED_LESSONS=1` is set. It runs:

1. `MemoryStore.getCandidates({ types: ['lesson', 'decision'], workdir, token_budget: 800, query: task })`
2. Sorts: failure-tagged lessons first, then by relevance
3. Prefixes `[UNVERIFIED]` for unverified, ` FAILED:` for failure-tagged
4. Returns a markdown block titled `## Recalled Lessons`

Preview what a worker would see:
```
relay memory show-context 'how do I fix the test'
```

This is the gap-bridge: workers (cheap LM Studio models, frontier OpenRouter) get the same hard-won lessons your CC sessions accumulated.

## SessionStart hook for Claude Code

Install a hook so every new CC session auto-injects recalled lessons:
```
relay memory hook --install
```

This writes to `<project>/.claude/settings.json` SessionStart array. The hook runs `relay memory recall` with the last commit message as the query. If the recall returns content, CC sees it as a system reminder before responding.

Uninstall:
```
relay memory hook --uninstall
```

## Promoting a memory to permanent rules

For extra-important memories that should be loaded as static project rules:
```
relay memory to-rules <memory_id> --rules-file .claude/CLAUDE.md
```
Appends the entry to the specified rules file under a `## Promoted Memory Rules` section.
